# v5.2 Papercompany Runtime 실전 검증

## 검증 대상

`insightflo/papercompany-runtime`의 고정 커밋 `b5f9caac6a4a9013f6e0abfd409c4847f6e04b5e`를 GitHub 원격 snapshot 모드로 분석했다. 25개 파일의 commit SHA, blob SHA, 줄 범위와 route/service/test/schema 교차 근거를 기록했다.

## 발견한 스킬 결함

1. **원격 connector를 불필요하게 저하 모드로 취급**: 고정 커밋과 blob SHA가 있어도 CodeGraph가 없으면 정상 분석으로 인정하지 않았다.
2. **Focus context summary가 자체 word budget을 위반**: 여러 실패 이유를 합친 카드가 `max_card_words`를 넘을 수 있었다.
3. **commerce 예제 누출**: 모든 Focus 그림에 주문·재고·결제 phase와 order/product/payment 아이콘이 하드코딩되어 있었다.
4. **도메인 지도의 고정 좌표·문구**: `Ordering` 중심의 6개 카드 배치를 임의의 코드베이스에도 적용했다.
5. **Atlas label 공간 부족**: 상태 전이와 계약 라벨이 카드 위에 겹쳤다.
6. **Focus/Atlas Canvas 경로 혼합**: map-model의 첫 drawing path가 Focus이면 Atlas Canvas가 존재하지 않는 `atlas/excalidraw/focus-*`를 가리켰다.
7. **Atlas 탐색 버튼의 목적지 구분 실패**: 서로 다른 상태·목차 문서가 모두 `Deep Atlas`라는 같은 표시명으로 렌더링됐다.
8. **상태 분기를 순차 흐름처럼 오해시킴**: 모든 상태를 한 행에 놓아 `active → completed` 화살표가 `paused` 카드 뒤를 지나면서 `active → paused → completed`처럼 보일 수 있었다.

## 수정 결과

- `remote-connector-snapshot`을 immutable evidence 정상 모드로 추가했다.
- Focus summary를 first-view word budget 안에서 자르되 source refs와 Atlas 링크는 유지한다.
- phase 제목·부제·아이콘을 visual model에서 파생한다.
- domain overview는 주 계약 사슬, 지원 경계, 저장 기반을 노드 수에 맞춰 배치한다.
- Atlas 상태·흐름 카드 간격을 diagram type에 맞게 넓혔다.
- Atlas Canvas는 supplied visual model의 business-stream child를 source ref로 찾아 연결한다.
- 탐색 버튼은 Atlas 목차·그림 목차·Mission/Run/Step/Issue 상태처럼 실제 목적지를 구분해 표시하고, 같은 표시명이 서로 다른 링크를 가리키면 검증 실패로 처리한다.
- 상태 머신은 정상 진행 spine을 위 행에, pause·failure·skip·cancel·blocked 같은 분기 상태를 아래 행에 배치해 직접 전이를 다른 상태의 순차 경유로 오해하지 않게 했다.

## 검증 결과

Papercompany 산출물에서 map, Focus, Atlas, Excalidraw, Obsidian, Canvas 검증은 오류 0으로 통과했다. 저장소의 테스트 코드는 근거로 읽었지만 실행 환경에 로컬 checkout이 없어 test suite 자체는 실행하지 않았다.
